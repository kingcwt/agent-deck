# Diff Review

## Overview

在相信当前编辑器文件的 git 改动之前先做审阅。解释这个文件里的每个改动点做了什么、为什么改、依赖哪些周边引用和项目上下文、是否有必要，以及这次改动是否保持了最小且有依据的范围。

## Workflow

### 1. Locate the review target

- 把 `/df` 和 `/diff-review` 视为这个工作流的显式快捷触发词。
- 默认把当前编辑器里打开或当前选中的文件视为审阅目标。
- 每次调用只审一个文件，不能自动切到别的改动文件，也不能扩展成整个仓库的多文件审阅。
- 先检查这个文件自己的 git diff；如果这个文件没有 git 改动，就直接说明并停止。
- 其他文件只能作为理解当前文件改动的辅助上下文，不能变成额外审阅对象。

### 2. Read context before judging

- 不能只看 diff hunk，必须阅读完整变更文件。
- 继续阅读理解每个改动点所必需的最近邻代码，比如导入的工具函数、相邻函数、附近类型定义、调用点，以及上下游引用位置。
- 优先复用项目已有约束，包括仓库规则、代码注释和本地约定，不要凭空发明审阅标准。
- 如果只看 patch 无法做出可靠判断，就不能直接下结论，必须结合上下文。

### 3. Explain the change

- 要把当前文件 diff 拆成若干有意义的改动点来讲，不能把整个文件只当成一个笼统改动。
- 对每个改动点，都要用具体行为说明它到底改变了什么。
- 对每个改动点，都要解释作者为什么可能会这样改，以及它想解决的问题是什么。
- 对每个改动点，都要结合周边引用和项目上下文说明“这个改动为什么成立”。
- 点明每个改动点影响的是业务逻辑、数据流、UI 行为、错误处理、性能、共享接口，还是仅仅局部可读性。
- 区分“事实”和“推断”；如果原因不是代码里明确写出的，要说明这是你基于代码和上下文做出的推断。

### 4. Judge necessity

- 对每个有实际意义的改动，判断它属于：
  - 对当前需求是必要的
  - 合理但不是唯一做法
  - 可疑、偏题或无关
- 检查相同目标是否能通过更小、更局部的改动完成。
- 检查它是否优先复用了现有 helper、类型、组件或既有模式，而不是额外引入新逻辑。
- 重点标记那些没有明确需求却扩大影响面的改动，尤其是公共工具、共享包、通用配置、schema 相关代码和大范围重构。

### 5. Check minimal-change discipline

- 明确检查是否存在以下越界迹象：
  - 与需求无关的清理或格式化噪音
  - 顺手优化
  - 其实不需要却改了共享代码
  - 本可复用却重复实现
  - 扩大接口导致下游被迫联动修改
- 如果改到了公共或共享模块，要解释为什么这件事有风险，以及是否存在局部替代方案。
- 如果较大的改动面确实合理，要明确说明“为什么必须这样改”。

### 6. Report clearly

- 如果发现不必要、有风险或理由不充分的改动，先按严重程度列出问题。
- 然后给出一段紧凑总结，覆盖：
  - 当前审阅的是哪个文件
  - 文件里有哪些关键改动点
  - 每个改动点为什么会存在
  - 每个改动点依赖哪些引用关系或项目上下文
  - 每个改动点是否必要
  - 整体改动范围是否最小
- 必要时给出清晰结论，例如 `necessary`、`acceptable but could be smaller` 或 `should be revised`。

## Guardrails

- 这是一个“审阅与解释优先”的工作流，不是默认直接改代码的工作流。
- 除非用户在审阅之后明确要求修复，否则不要静默修改代码。
- 优先给出具体文件和行号，而不是泛泛而谈。
- 不要把它变成多文件审阅；真正被审的对象始终只有当前文件的 git diff。
- 如果改动涉及共享代码，除非你能解释为什么局部修改行不通，否则不要轻易判定这种改动是合理的。
- 如果没有可审阅的 diff，就直接说明并停止，不要编造审阅结果。
